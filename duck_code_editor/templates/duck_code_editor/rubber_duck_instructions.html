PART 3: Rubber Duck Debugging

This is the final part of the experiment. We will now introduce the concept of rubber duck debugging. Please read the following text carefully.

Rubber duck debugging is a technique where a programmer explains the code which they are working on to an inanimate object (usually a rubber duckie). The idea is, then, that by explaining the code as one goes through it, it will become easier to spot mistakes. For example:

GIVEN CODE: d = a + b * c

CODER: “And here d is initiated to the value of a + b, and that is multiplied by c.”

RUBBER DUCK: “...”

CODER: “Oh gosh! You’re right! There are no parentheses so b * c is happening first!

We would now like to ask you to use rubber duck debugging in the final tasks. We will provide you with a virtual rubber duckie on your screen. Please state ALL your thoughts out loud. Do not just think your thoughts, but explicitly voice them. Even though there will also be a 10 minutes limit per task, focus on using the technique as extremely and thoroughly as possible. For the next tasks, the rubber duckie is your best friend. You are not being recorded.

For the next tasks, please work only with rubber duck debugging!
