Part 3: Rubber Duck Debugging

This is the final part of the experiment. We will now introduce the concept of rubber duck debugging. Please read the following text carefully, as it will explain how to tackle the final task.

Rubber duck debugging is a technique where a programmer explains the code which they are working on to an inanimate object (usually a rubber duckie). The idea is, then, that by explaining the code as one goes through it, it will become easier to spot mistakes. For example:

GIVEN CODE: d = a + b * c

CODER: “And here d is initiated to the value of a + b, and that is multiplied by c.”

RUBBER DUCK: “...”

CODER: “Oh gosh! You’re right! There are no parentheses so b * c is happening first!

We would now like to ask you to use rubber duck debugging in the final task. We will provide you with a virtual rubber duckie on your screen. Please state ALL your thoughts out loud. Do not just think your thoughts, but explicitly voice them. Really try to use the technique as extremely and thoroughly as possible. For the next ten minutes, the rubber duckie is your best friend. You are not being recorded.

For the next task, please work only with rubber duck debugging!
